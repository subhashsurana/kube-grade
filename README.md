# kube-grade

**Self-grading scripts for Kubernetes certification exams** вҖ” know your answer is correct before hitting the Killercoda or killer.sh check button.

![CI](https://github.com/kube-grade/kube-grade/actions/workflows/ci.yml/badge.svg)
![Version](https://img.shields.io/github/v/tag/kube-grade/kube-grade?label=version)
![License](https://img.shields.io/badge/license-MIT-green)

---

## Why

Killercoda's check button is a black box. You apply your YAML, get 0 points, and have no idea which field was wrong. kube-grade runs the same `kubectl` jsonpath checks the grader uses вҖ” with colour-coded PASS/FAIL output and the exact command printed for every check вҖ” so you fix the real problem instead of guessing.

---

## Install

One command, works in any Killercoda or killer.sh terminal:

```bash
curl -sL https://raw.githubusercontent.com/kube-grade/kube-grade/main/install.sh | bash && source ~/.bashrc
```

## Usage

### Generate a grader for the task you're solving

```bash
new_task 3 ckad     # вҶ’ ~/kube-grade/ckad/tasks/task3-grade.sh
new_task 7 cka      # вҶ’ ~/kube-grade/cka/tasks/task7-grade.sh
new_task 2 cks      # вҶ’ ~/kube-grade/cks/tasks/task2-grade.sh
```

Edit the `CONFIG` section at the top вҖ” names, namespace, image вҖ” uncomment the `check_` lines that match the task's bullet points, then:

```bash
bash ~/kube-grade/ckad/tasks/task3-grade.sh
```

### Run a pre-built scenario grader

```bash
kg ckad pod           # grade a basic CKAD pod task
kg ckad rbac          # grade an RBAC task
kg cka  etcd-backup   # grade a CKA etcd backup task
kg cks  seccomp       # grade a CKS seccomp task
```

Override config with env vars вҖ” no editing needed:

```bash
POD_NAME=web-pod NS=staging IMAGE=nginx:1.19 kg ckad pod
```

### Load the library directly for one-off checks

```bash
source ~/kube-grade/lib/grade-lib.sh

check_exists      pod "web-pod" "staging"
check_pod_ready       "web-pod" "staging"
check_image           "web-pod" "staging" "nginx:1.19"
grade_summary
```

## Check functions reference

| Function | What it checks |
|---|---|
| `check_exists KIND NAME [NS]` | Resource exists |
| `check_namespace NS` | Namespace exists |
| `check_pod_ready NAME [NS]` | Pod Running + all containers Ready |
| `check_status KIND NAME NS EXPECTED` | .status.phase |
| `check_image NAME NS IMAGE [IDX]` | Pod container image |
| `check_deploy_image NAME NS IMAGE [IDX]` | Deployment template image |
| `check_label KIND NAME NS KEY VAL` | metadata.labels key=value |
| `check_selector_label KIND NAME NS KEY VAL` | spec.selector.matchLabels |
| `check_replicas NAME NS N` | Deployment replicas |
| `check_env POD NS KEY VAL` | Live env var (via kubectl exec) |
| `check_env_from_configmap POD NS IDX KEY CM KEY` | Env sourced from ConfigMap ref |
| `check_env_from_secret POD NS IDX KEY SEC KEY` | Env sourced from Secret ref |
| `check_volume_mount POD NS PATH [VOL]` | volumeMount path |
| `check_secret_key SECRET NS KEY DECODED_VAL` | Secret key decoded value |
| `check_service_type SVC NS TYPE` | Service type |
| `check_service_port SVC NS PORT [TARGET]` | Service port + targetPort |
| `check_service_selector SVC NS KEY VAL` | Service selector |
| `check_service_endpoints SVC NS` | Service has вүҘ1 active endpoint |
| `check_resources POD NS CPU_REQ MEM_REQ CPU_LIM MEM_LIM` | Resource requests/limits |
| `check_probe POD NS TYPE [PATH] [PORT] [DELAY]` | liveness/readiness/startup probe |
| `check_pvc NAME NS [SIZE] [ACCESS] [SC]` | PVC Bound + storage/access/class |
| `check_rbac SA NS VERB RESOURCE [EXPECTED]` | kubectl auth can-i |
| `check_rolebinding_role RB NS ROLE` | RoleBinding roleRef |
| `check_rolebinding_subject RB NS SA` | RoleBinding subject |
| `check_cronjob NAME NS [SCHEDULE] [IMAGE]` | CronJob schedule + image |
| `check_ingress NAME NS HOST PATH SVC PORT` | Ingress host/path/backend |
| `check_netpol NAME NS [KEY] [VAL]` | NetworkPolicy + podSelector |
| `check_file_exists PATH` | File exists |
| `check_file_contains PATH SUBSTRING` | File contains string |
| `check_file_exact PATH CONTENT` | File exact content |
| `check_jsonpath KIND NAME NS JP EXPECTED [DESC]` | Any jsonpath field |
| `grade_summary` | Print score + failed checks; exits non-zero if any failed |
| `grade_reset` | Reset counters between independent tasks |

## Recommended workflow

**Write the grader before you solve the task.** Read the task description, translate every bullet point into a `check_` call in your `task3-grade.sh`, then solve. When you run the grader after applying your YAML, each failure tells you the exact field and the exact mismatch вҖ” no guessing.

1. `new_task 3 ckad` вҖ” generate the grader
2. Edit CONFIG + uncomment relevant checks
3. Solve the task, apply your YAML
4. `bash ~/kube-grade/ckad/tasks/task3-grade.sh`
5. Fix failures, re-run until `3/3 (100%)`
6. Hit Killercoda Check вҖ” already know the answer
